## 왜 브랜치 전략이 중요한가

혼자 개발할 때는 main 브랜치 하나로도 충분합니다. 하지만 팀원이 2명만 되어도 코드 충돌이 발생하기 시작합니다. 누군가 로그인 기능을 개발하는 동안 다른 사람이 결제 기능을 만들면, 서로의 코드가 엉키는 상황이 생깁니다.

브랜치 전략은 이런 문제를 해결하기 위한 약속입니다. 언제 브랜치를 만들고, 어떻게 이름 짓고, 언제 합칠지에 대한 규칙을 정하는 것이죠.

## Git Flow

Vincent Driessen이 2010년에 제안한 전략으로, 가장 잘 알려진 브랜치 모델입니다.

### 브랜치 구조

- **main**: 배포된 코드만 존재
- **develop**: 다음 릴리즈를 위한 개발 브랜치
- **feature/**: 기능 개발용 (develop에서 분기)
- **release/**: 릴리즈 준비용 (develop에서 분기)
- **hotfix/**: 긴급 버그 수정용 (main에서 분기)

### 장점

정해진 규칙이 명확해서 대규모 팀에서 혼란 없이 사용할 수 있습니다. 릴리즈 주기가 정해진 프로젝트에 적합합니다. 버전 관리가 체계적이어서 어떤 버전에 어떤 기능이 포함되었는지 추적하기 쉽습니다.

### 단점

브랜치가 많아서 복잡합니다. CI/CD와 함께 사용하면 머지가 빈번해지고, 충돌 해결에 시간이 많이 걸릴 수 있습니다. 웹 서비스처럼 지속적으로 배포하는 프로젝트에는 과한 구조입니다.

### 적합한 경우

- 모바일 앱처럼 버전 릴리즈가 있는 프로젝트
- 10명 이상의 대규모 팀
- QA 프로세스가 별도로 있는 조직

## GitHub Flow

GitHub에서 사용하는 간소화된 전략입니다. main 브랜치와 feature 브랜치만 사용합니다.

### 워크플로우

1. main에서 새 브랜치 생성
2. 코드 작성 및 커밋
3. Pull Request 생성
4. 코드 리뷰
5. main에 머지
6. 즉시 배포

### 장점

단순합니다. 브랜치 두 종류만 이해하면 됩니다. PR 기반으로 코드 리뷰가 자연스럽게 이루어집니다. CI/CD와 궁합이 좋아서 머지하면 바로 배포되는 파이프라인을 만들기 쉽습니다.

### 단점

복잡한 릴리즈 관리가 어렵습니다. 여러 버전을 동시에 유지해야 하는 프로젝트에는 부적합합니다. main이 항상 배포 가능한 상태여야 하므로 테스트 자동화가 필수입니다.

### 적합한 경우

- 웹 서비스, SaaS 프로젝트
- 소규모 팀 (2~5명)
- 지속적 배포를 하는 프로젝트

## Trunk-Based Development

모든 개발자가 하나의 브랜치(trunk/main)에 직접 커밋하는 방식입니다. Google, Facebook 등 대형 IT 기업에서 사용합니다.

### 핵심 원칙

- 브랜치 수명은 최대 1~2일
- 하루에 여러 번 main에 머지
- Feature Flag로 미완성 기능 숨김
- 강력한 테스트 자동화 필수

### 장점

머지 충돌이 최소화됩니다. 코드가 항상 통합된 상태이므로 "통합 지옥"이 없습니다. 배포 주기가 매우 빠릅니다.

### 단점

높은 수준의 테스트 자동화와 CI/CD가 전제조건입니다. Feature Flag 관리 비용이 추가됩니다. 경험이 부족한 팀에서는 main이 불안정해질 수 있습니다.

## 어떤 전략을 선택할까

팀 규모와 프로젝트 특성에 따라 달라집니다.

**1~3명 소규모 팀**: GitHub Flow를 추천합니다. 간단하고 빠르게 적용할 수 있습니다.

**5~15명 중규모 팀**: GitHub Flow에 릴리즈 브랜치만 추가하는 변형을 추천합니다. 복잡하지 않으면서도 릴리즈 관리가 가능합니다.

**15명 이상 대규모 팀**: Git Flow나 Trunk-Based 중 선택합니다. 릴리즈 주기가 있으면 Git Flow, 지속적 배포면 Trunk-Based가 적합합니다.

## 실무 팁

### 커밋 메시지 컨벤션

브랜치 전략만큼 커밋 메시지도 중요합니다. Conventional Commits를 추천합니다.

- `feat:` 새 기능
- `fix:` 버그 수정
- `docs:` 문서 수정
- `refactor:` 리팩토링
- `test:` 테스트 추가

### 브랜치 이름 규칙

팀에서 통일된 규칙을 사용하세요.

- `feature/로그인-기능` 또는 `feature/JIRA-123-로그인`
- `bugfix/결제-오류-수정`
- `hotfix/보안-취약점`

### PR 템플릿 활용

GitHub에서 `.github/PULL_REQUEST_TEMPLATE.md`를 만들면 PR 작성 시 자동으로 템플릿이 적용됩니다. 변경사항 요약, 테스트 방법, 스크린샷 등의 항목을 포함하면 리뷰 품질이 올라갑니다.

## 마무리

완벽한 브랜치 전략은 없습니다. 중요한 것은 팀원 모두가 같은 규칙을 이해하고 따르는 것입니다. 처음에는 GitHub Flow처럼 간단한 전략으로 시작하고, 필요에 따라 규칙을 추가해 나가는 것이 현실적인 접근법입니다.
